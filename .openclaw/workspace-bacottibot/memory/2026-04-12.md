## Reflection: Bithues Bulk Fix Session (2026-04-12 Evening)

### What Went Well
- **Link audit subagent** was fast and thorough — found 78+ real issues (199 broken fiction.html links, 42 breadcrumb path errors, accent mismatch, canonical issues, orphan directory). Good use of a subagent.
- **Fiction.html fix** (199 links → catalog.html) was clean and fast — done in 1 minute
- **Incremental push discipline** on successionholdingllc.com (canonical fix + sitemap + robots.txt all pushed cleanly in one commit)
- **Cloudflare token issue** handled correctly — no workspace secrets found, clear explanation to Mike

### Mistakes Made
1. **Subagent reports weren't verified before acting on them** — the articles fixer reported "article body missing" (status: timeout) but actual content was there. Should have checked the live site first.
2. **Bulk `<style>` removal broke story pages** — the first-pass cleanup script was too aggressive with `sed 's/<style>.*/<style>/'` patterns, destroying nav blocks and content. The `.*` in sed is greedy and matches across lines incorrectly.
3. **Parallel subagents on overlapping filesets caused wasted work** — 5 subagents running simultaneously on reviews/, stories/, articles/, authors/, and general fixes meant they couldn't see each other's changes. Some work was duplicated.
4. **Timeout wasted tokens** — stories fixer (10min, 1.3M tokens) and articles fixer (10min, 117K tokens) both timed out without completing. The work they did was mostly already done by earlier subagents.
5. **Should have pushed to GitHub incrementally** — 183 files in one commit caused a fetch-first conflict on push. Would have been smoother to commit after each subagent finished.
6. **Didn't check live site structure before spawning all fixers** — spawned 4 parallel structural fixers without first verifying what the actual current state was.

### Key Lessons Learned
1. **Always verify before acting on subagent reports** — especially timeout reports. Check the actual files/live site before spawning follow-up work.
2. **Bulk HTML manipulation with sed/perl is fragile** — greedy regex patterns like `<style>.*</style>` break on multiline content. Better approach: use proper parsers or very targeted single-line fixes.
3. **Subagents working on the same directory should be sequential, not parallel** — or at minimum, wait for one to finish before spawning the next.
4. **Push to GitHub after each subagent completes** — don't batch 5 subagents worth of changes and try to push all at once.
5. **Check live site structure BEFORE spawning fixers** — the homepage structural audit (nav/hero/chips/article/footer) took 10 minutes but could have been done in 2 minutes with `curl | grep`.
6. **Bithues has 3 page types with different structure needs**:
   - Homepage: nav → hero → chips → sections → footer
   - Review/Story pages: nav → hero → chips → article → share → footer
   - Article pages: nav → hero → chips → article → share → footer
   - All need: `<link rel="stylesheet" href="../css/main.css">` in subdirectories
