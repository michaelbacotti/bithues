## Morning Session Issues — 2026-04-07

**What happened:** Around 08:30 EDT, assistant stopped responding mid-conversation. Mike had to diagnose and fix by:
1. Speaking with a subagent that restarted the gateway
2. Deleting subagents and a cron job

**Root causes identified (this morning):**
1. **Did work directly instead of via subagent** — when Mike asked for the market forecast, assistant did web research and page updates directly in the main session. This blocks the conversation, uses main session tokens for work, and if the session has issues the work fails too.
2. **Long conversation context** — context window grew too large, causing memory degradation and slow/unresponsive tool calls
3. **Subagent timeout pattern** — subagents consistently timing out on multi-step GitHub API workflows

**What Mike said (important):**
- "you did work instead of having your subagents do the work which made things worse"
- "is this conversation too long?"
- "evaluate what happened"

**Fixes applied:**
- Updated `skills/conversation-hygiene/SKILL.md` with explicit "use subagents" rule
- Rule: ANY substantive task → spawn subagent immediately
- Quick status checks → only do directly
- After any main session issue, write checkpoint to memory before continuing

**New rule going forward:**
- If a task takes more than 2 tool calls, it should be a subagent
- If Mike asks you to "update the website", "investigate", "check", "fix" → spawn subagent
- Never block the main session with work
